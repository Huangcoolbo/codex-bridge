# codex-bridge 项目设计说明（第一版）

[English Version](./PROJECT_DESIGN.md)

## 1. 项目目标

`codex-bridge` 的目标不是自己替用户分析远程电脑上的问题，而是作为一座“桥”，把**本地 Codex 的操作能力延伸到远程电脑**。

它的核心定位是：

- **本地 Codex 负责思考、判断、决定下一步动作**
- **桥负责把动作传到远程电脑，并把执行结果带回来**
- **远程电脑负责真正执行这些动作**

第一阶段重点支持：

- 本地 Codex 通过桥操作远程 Windows
- 远程 Windows 通过 PowerShell 执行命令
- 后续逐步扩展到 Linux 和 Android

---

## 2. 核心设计思想

这个项目解决的不是“远程文件同步”，也不是“远程桌面”。

它解决的是：

> 如何让本地运行的 Codex，像把手伸到远程电脑上一样，连续地下达动作并获取结果。

所以这个项目的核心思想是：

- **脑子在本地**
- **动作在远程**
- **桥负责连接这两边**

桥的第一职责不是“理解任务”，而是：

- 传递命令
- 返回结果
- 保持调用方式统一

---

## 3. 三个核心模块

### 模块一：本地 Codex

职责：

- 接收用户任务
- 决定下一步做什么
- 组织多步操作
- 根据远程结果继续判断

本地 Codex 是“脑子”，它不直接操作远程系统，而是通过桥发出动作。

---

### 模块二：桥（remote-agent-bridge）

职责：

- 管理远程主机信息
- 建立本地到远程的连接通道
- 把本地动作传到远程执行环境
- 接收远程执行结果
- 以统一格式返回结果

桥不是执行者，也不是分析者。桥的本质是：

> 让本地 Codex 能稳定、连续地操作远程电脑。

---

### 模块三：远程执行端

第一阶段中，远程执行端直接使用：

- **远程 Windows 的 PowerShell**

职责：

- 接收桥送来的命令
- 在远程电脑上实际执行
- 访问远程文件、目录、进程、服务等资源
- 返回输出、错误信息和执行状态

在第一版里，不额外开发远程常驻代理程序。

原因是：

- PowerShell 已经具备最基础的执行能力
- 可以减少远程端部署复杂度
- 更适合先验证整体方案

后续如果需求变复杂，再考虑把第三模块升级成专门的远程执行代理。

---

## 4. 第一阶段的目标能力

第一阶段必须优先完成的是“远程执行基础动作”。

### 第一优先级

- 远程执行 PowerShell 命令
- 返回标准输出
- 返回错误输出
- 返回执行状态

### 第二优先级

在第一优先级稳定后，再补：

- 读取远程文件
- 列出远程目录
- 写入远程文件
- 搜索文本
- 获取系统信息

### 第三优先级

在上面稳定后，再考虑：

- 连续操作上下文
- 远程工作目录概念
- 批量动作
- 更清晰的结果结构
- 更强的安全控制

---

## 5. 第一阶段不做什么

为了避免项目一开始就过重，第一阶段明确不做这些：

- 不做远程桌面
- 不做 GUI 自动化
- 不在远程电脑上部署额外 agent
- 不把复杂分析逻辑塞进桥里
- 不把桥做成“大而全”的平台

第一阶段只做一件事：

> 让本地 Codex 能可靠地把动作打到远程 Windows 上。

---

## 6. 当前代码结构与职责分工

### 启动与命令入口

- `__main__.py`：启动程序
- `cli.py`：接收命令并分发请求

### 调度与数据

- `service.py`：协调整个流程
- `models.py`：定义主机资料和结果格式
- `storage.py`：保存和读取主机资料
- `exceptions.py`：统一错误表达

### 装配逻辑

- `factory.py`：根据主机类型和连接方式，决定该创建哪种执行组合

### 连接层

- `adapters/base.py`：连接层共同规则
- `adapters/ssh.py`：通过 SSH 建立到远程 Windows 的通道

### 平台层

- `providers/base.py`：平台能力共同规则
- `providers/windows.py`：把远程 Windows 操作转换成具体执行动作

## 7. 一次典型调用的数据流

以“查看远程 OpenClaw.json”为例，理想数据流是：

1. 用户给本地 Codex 一个任务
2. 本地 Codex 决定先检查远程路径或读取文件
3. Codex 通过桥发出远程动作
4. 桥把动作通过连接层送到远程 Windows
5. 远程 PowerShell 执行动作
6. 执行结果返回给桥
7. 桥把结果交回本地 Codex
8. Codex 继续决定下一步动作

这个过程的关键点是：

- 判断在本地
- 执行在远程
- 连接由桥完成

---

## 8. 为什么第一阶段选择 PowerShell

选择远程 PowerShell 作为第一阶段执行端，原因有几个：

- Windows 原生自带
- 能力足够覆盖第一阶段需求
- 可以直接做文件、目录、进程、服务等操作
- 不需要在远程机再安装一个新的常驻程序
- 更利于尽快验证“本地 Codex 破界”这件事

所以第一阶段里：

> 第三模块可以理解成“远程 PowerShell 执行层”。

但严格来说，它不是一个新写的独立程序，而是利用远程系统已有能力来承担执行职责。

---

## 9. 后续扩展方向

在第一阶段验证成功后，可以往几个方向扩展：

### 方向一：平台扩展

- 支持 Linux
- 支持 Android

### 方向二：能力扩展

- 上传文件
- 下载文件
- 文件修改
- 服务控制
- 日志抓取

### 方向三：执行模型扩展

- 保持会话上下文
- 增加动作队列
- 增加更强的结果结构化能力
- 增加安全策略和审批机制

### 方向四：远程执行端升级

当 PowerShell 方式不够用时，再考虑：

- 开发远程常驻执行代理
- 降低重复连接和重复启动开销
- 提供更稳定的持续操作能力

---

## 10. 第一阶段完成标准

第一阶段只有满足下面这些条件，才算真正完成：

- 本地能稳定配置远程 Windows 主机
- 本地能通过桥向远程 Windows 发起命令
- 远程 Windows 能执行命令并返回结果
- 本地 Codex 能基于这些结果继续下一步判断
- 至少完成真实远程场景的一次完整验证

如果只是“代码已经写了”，但没有把这条链路跑通，就不能算完成。

---

## 11. 一句话总结

`remote-agent-bridge` 第一阶段的真正目标是：

> **把本地 Codex 的操作能力，通过桥延伸到远程 Windows，让远程 PowerShell 成为第一版执行端。**
