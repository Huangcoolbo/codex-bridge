# 工程路线图

这份文档不讲产品愿景，而是讲：

```text
从现在这版代码
  ->
到“真实可用”的桥接工具
  ->
还差哪些工程工作
```

这里的“真实可用”不是：

- 本机能跑一次
- 作者设备能跑
- 某条路径演示成功

而是：

```text
真实 Windows 主机
真实 Android 设备
连续任务
失败可恢复
结果可信
```

## 1. 当前判断

当前项目已经具备：

- agent-first + gateway-first 主方向
- Windows SSH + PowerShell 主路径
- Android ADB 主路径
- 本地 HTTP gateway
- 桌面客户端作为配置与诊断壳

但离“长期稳定可信”还差最后一段工程打磨。

可以粗略理解成：

```text
现在
  ->
已经能用

目标
  ->
敢长期依赖
```

## 2. 当前最重要的 5 个短板

### 2.1 真实设备验证不够

现在很多能力是：

```text
代码支持
  !=
真实异构设备稳定可用
```

尤其需要补：

- 不同 Windows 主机
- 不同 Android 品牌 / ROM / 版本
- 不同网络环境

### 2.2 Android 无线链路还不够稳

当前 Android 无线路径已经能：

- pair
- connect
- 学习并回填 connect endpoint

但还需要继续收敛：

- mDNS 发现不稳定时的恢复
- 配对失效和地址失效的区分
- 真机差异验证

### 2.3 Gateway 安全模型还只是起步

现在已经补了：

- token 鉴权
- 审计日志

但仍然只是最小边界，还缺：

- 更清楚的调用方身份
- 更细的读 / 执行 / 写权限
- 更危险动作的策略面

### 2.4 测试深度不够

现在测试主要还在：

- 路由匹配
- gateway 鉴权核心
- 一部分 Android / service 行为

还缺：

- 路由级集成测试
- Windows / Android 关键行为回归
- 更接近真实调用链的测试

### 2.5 长任务与恢复机制不够

当前系统更像：

```text
发请求
  ->
等结果
  ->
写日志
```

未来更真实的形态应该更接近：

- 长任务状态
- 超时与重试
- 流式输出
- 取消
- 失败后恢复

## 3. 必做项

这些事不做，项目很难从“能用”迈到“可信”。

### 3.1 打满真实联调

优先顺序：

```text
Windows 真远端
  ->
Android USB 真机
  ->
Android 无线真机
```

至少要把这些动作在真实设备上跑通并记录：

- probe
- execute
- list
- read
- controlled write

### 3.2 收敛 Android 无线状态机

当前最需要继续做的是：

- 区分：
  - 已配对
  - 已发现连接地址
  - 已连接
  - 地址失效
  - 需要重新配对
- 让失败提示更接近真实原因

### 3.3 补齐 gateway 集成测试

优先补这几类：

- `/health`
- `/api/targets/*`
- `/api/probe`
- `/api/command/execute`
- `/api/android/*`

重点不是测 JSON 外壳，而是测：

```text
鉴权
  ->
路由
  ->
service
  ->
结果结构
```

### 3.4 清理当前本地遗留物

当前工作区仍有中途实验或未整理内容，例如：

- `desktop-client/src/shared/androidPairing.ts`

这类文件如果不及时清理，会让：

- 当前主流程
- 仓库历史
- skill 叙事

逐渐不一致。

## 4. 应做项

这些不是第一刀，但会明显提升真实可用性。

### 4.1 Gateway 权限继续细化

下一阶段可以考虑：

- 只读 vs 执行 vs 写
- target 级别限制
- 更明确的调用方身份

### 4.2 Android 文件与 shell 差异收敛

重点补：

- toybox / grep / stat 差异
- base64 行为差异
- 权限路径差异
- 编码与换行差异

### 4.3 更明确的操作审计

当前审计已经有了基础壳。

后面更值得补的是：

- 谁发起的
- 发起到哪个 target / device
- 做了什么动作
- 结果如何

## 5. 可后做项

这些现在不是主战场。

### 5.1 Linux 路线

现在不该让 Linux 分散主战场。

顺序应该是：

```text
先把 Windows / Android 打稳
再谈 Linux
```

### 5.2 更复杂的客户端表现层

客户端现在已经够承担：

- 配置
- 状态
- 结果
- 诊断

在内核稳定之前，不应该优先把大量时间放到：

- 更复杂的动效
- 更重的视觉系统
- 更大范围的 UI 扩张

## 6. 路线图顺序

最推荐的顺序是：

```text
1. 真机 / 真远端联调
2. Android 无线状态与恢复收敛
3. gateway 集成测试补强
4. 审计与权限继续细化
5. 长任务 / 流式输出 / 取消
6. Linux 或更大平台扩展
```

## 7. 一句话总结

这项目现在最缺的，不是再多做几个功能点，而是：

```text
把已有能力
从“写出来”
推进到“真实环境里长期稳定”
```

后面最值钱的工作，也会主要是这部分工程厚度，而不是新概念。
